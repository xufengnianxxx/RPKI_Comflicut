# 项目实现报告  
## 基于区块链的 RPKI 证书资源冲突校验系统的设计与实现

---

### 1. 项目总体目标与最终实现效果

#### 1.1 生动总述

本系统最终呈现为一套 **「去中心化的 RPKI 自动巡查与产权登记系统」**：链下服务如同 **7×24 小时的数字地籍巡查队**，从各区域互联网登记机构（RIR）发布的仓库包或本地目录快照中 **批量解析数千张 RPKI 证书**，在 **IPv4/IPv6 地址块与 AS 号** 构成的资源平面上执行 **形式化冲突检测**——发现「地界重叠」「继承越权」「授权自相矛盾」等问题后，立即在 **MySQL** 中形成结构化台账（冲突对、类型、严重度、规则引用）。随后，系统如同 **多方共同见证的登记处**，将 **脱敏后的证书摘要与冲突证据摘要** 提交至 **Hyperledger Fabric** 账本，由 **Java 智能合约** 按 **确定性谓词** 执行 **链上核心校验** 与 **不可篡改存证**。第三方可仅凭链码与历史交易 **独立复核** 结论；应用侧通过 **`fabric_tx_record` 与 `correlation_id`** 记录 **检测→上链** 全过程。若 Fabric 暂时不可用，**链下检测结论不会被撤销**，系统以 **`fabric_send_failed` / `fabric_last_error`** 明示失败原因，体现 **可审计、可运维、最终一致** 的工程完整性。

#### 1.2 开题核心目标与实现对照表

| 开题常见目标 | 本系统实现要点 | 实现程度 |
|--------------|----------------|----------|
| **链上核心校验 + 链下辅助计算** | 链下：`RpkiUnifiedConflictDetectionService` 全量/可配置策略检测；链上：`detectConflictOnChain` / `detectConflictOnChainBatch`；集成：`FabricOnChainDetectSyncService` 回写与再存证。 | 已实现 |
| **形式化冲突建模与规则可验证** | 链码：`Prefix4`/`Prefix6` 闭区间语义、`cidrIntervalSubset` / `cidrIntervalsIntersect` / `rangesAdjacent`、`isASConflict`、`collectInheritanceViolations` 等与注释中数学表述一致；链下：`IPAddressUtils`、`Ipv6CidrUtils`、BigInteger 区间。 | 已实现 |
| **不可篡改存证与可审计追溯** | 账本键 `CERT|*`、`CONFLICT|*`；`queryConflicts`；`getAuditHistory`/`getHistoryForKey`（证书键历史）；应用表 `fabric_tx_record` + `correlation_id` + `pipeline_step`；链码 `[AUDIT]` 日志。 | 已实现 |
| **隐私与最小披露** | `CertificateRedactedPayload` / `CertificateAsset` 仅哈希与资源摘要等，非完整原始 DER。 | 已实现 |
| **异常一致性与失败可观测** | `fabric_send_failed`、`fabric_last_error`；`FabricConflictAnchorService.submitWithRetry`；链码 **`txTimestampEpochMs`（`getTxTimestamp`）** 保证多组织背书确定性。 | 已实现 |

| 持久化与分页查询 | `StorageService` | 实体、分页参数 | `Page`、列表 | 链下权威台账 |

#### 2.2 冲突检测层

| 功能 | 实现位置 | 输入 | 输出 | 创新点体现 |
|------|----------|------|------|------------|
| 统一检测主流程 | `RpkiUnifiedConflictDetectionService#detectAndPersistConflicts` | 可选 `rir` | `List<ConflictResult>` + 写库 | **多模块流水线** |
| 多跳继承/撤销 | `detectMultiHopChain`、`walkAncestors` | 有效证集合、`byId` | 追加 `ConflictResult` | **祖先已撤后代仍有效** |
| IPv4 桶索引成对检测 | `detectPairwiseIndexed`、`firstOctetOfCidr` | 同上 | 枚举对数 + 结果 | **O(n·桶规模) 近似** 替代全量 O(n²) |
| 成对 IP/AS/启发式 | `detectIpPair`、`detectAsPair`、`detectIpAsBindingHeuristic`、`detectDirectRevokedPair` | 两证 | 多条 `ConflictResult` | **RFC3779/6487/6811 思路对齐** |
| 父子与对等范围控制 | `isResourceParentForScope`、`isComparablePeerContext` | 两证 | boolean | **可配置研究参数**（`rpki.conflict.*`） |
| 兼容门面 | `ConflictDetectionService` → 委托统一引擎 | 同 API | 同输出 | API 稳定 |
| 冲突键 | `ConflictKeyBuilder` | id/类型/细节 | `conflict_key` 字符串 | **与链上键片段对齐** |

**链下无冲突时的链上行为（自动锚定开启时）**：`anchorAfterDetection` 在 **`results` 为空** 时 **立即返回**，**不调用** `storeCertificateBatch` / `recordConflictEvidenceBatch`，**不产生**本轮 `OFFLINE_DETECT_ANCHOR_*` 审计标记之前的提交类 `fabric_tx_record`（因提前 return）。链下仍会 **`resetConflictFlags`、`saveConflictRecords`（无新行）、`updateCertConflictSummary`**。

**链下有冲突时**：构造证书载荷与证据载荷 → **重试提交** 证书批次（若非空）与 **冲突批次** → 成功则回写 Fabric 字段；失败则 **`fabric_send_failed`** 等。

#### 2.3 链上层（智能合约 + Fabric）

| 功能 | 链码函数 | 输入 | 输出 | 创新点 |
|------|----------|------|------|--------|
| 批量存证证书 | `storeCertificateBatch` | JSON 数组 | 含 `txId` 的 JSON | 脱敏世界状态 |
| 双证检测 | `detectConflictOnChain` | `certHashA,B` | `ConflictDetectResult` JSON | **链上核心校验** |
| 批量检测 | `detectConflictOnChainBatch` | `DetectBatchRequest` JSON | 批量 JSON | 与链下同构批量复核 |
| 批量冲突存证 | `recordConflictEvidenceBatch` | 证据数组 JSON | 结果 JSON | **与 detect 命中写入同路径** |
| 只读查询 | `queryCertificate`、`queryConflicts` | 参数 | JSON | 展示与对账 |
| 键历史 | `getAuditHistory`、`getHistoryForKey` | `certHash` | JSON 数组 | **Fabric 原生审计** |

### 4. 关键代码文件与函数详细说明

#### 4.1 实体与数据库层

##### 4.1.1 `RpkiCert.java`（表 `rpki_cert`）

**类作用**：单张 RPKI 证书的 **主数据**、**链下冲突摘要**、**Fabric 回写与失败标记** 的统一载体。

| 字段分组 | 字段名 | 详细含义 | 在系统中的作用 |
|----------|--------|----------|----------------|
| 标识与展示 | `id` | 自增主键 | ORM、外键关联 `conflict_record.cert_id_*` |
| | `certName`、`cerFileName`、`serialNumber`、`issuer`、`subject`、`rir` | 人类可读与来源 RIR | 列表、调试、`CertificateRedactedPayload.certNameShort` |
| 有效期 | `notBefore`、`notAfter` | UTC 有效窗 | `filterValidAt` 过滤检测对象 |
| RFC 3779 | `ipv4Prefixes`、`ipv6Prefixes`、`asNumbers` | JSON 串存前缀/AS 列表 | 链下 **`detectIpPair`**、链上 **`CertificateAsset`** |
| 证书链 | `subjectKeyId`、`authorityKeyId`、`parentCertId`、`revoked` | SKI/AKI、父指针、撤销标记 | **`isResourceParentForScope`**、多跳撤销检测、脱敏上链 |
| 链下检测摘要 | `hasConflict` | 是否涉冲突（布尔） | 列表筛选、报表；**上链失败不强制改回 false** |
| | `conflictDetails` | 多冲突拼接说明 | 人类可读；锚定成功时可聚合刷新 |
| Fabric 成功态 | `fabricTxId` | 最近相关交易号（锚定成功时常为冲突批次 tx） | 与账本对账 |
| | `fabricBlockNum` | 预留区块号 | 可扩展 |
| | `isSentToFabric`、`fabricSendTime` | 是否已成功标记上链、时间 | 运维展示 |
| **Fabric 失败态** | **`fabricSendFailed`** | 重试耗尽后仍为 true | **失败可观测**；与 `hasConflict` 解耦 |
| | **`fabricLastError`** | 截断后的异常信息 | 论文「异常处理」章节直接引用 |
| 内容哈希 | `certHash` | 唯一；与 `CERT\|hash` 对齐 | 链上查询、DB 去重 upsert |
| 原始数据 | `rawCertData` | 可选存 DER/Base64 | 解析来源；一般不上链全文 |
| 审计 | `createdAt`、`updatedAt` | 记录时间 | 填充策略 |

##### 4.1.2 `ConflictRecord.java`（表 `conflict_record`）

**类作用**：**每一对证书、每一次规则命中** 一条记录，承载 **类型、严重度、规则版本引用**。

| 字段 | 含义 |
|------|------|
| `id` | 主键 |

**类作用**：应用侧 **审计流水**——弥补 **MySQL 与 Fabric 无法同一 ACID** 的过程证据。

##### 4.2.2 `RpkiUnifiedConflictDetectionService.java`（冲突检测主实现；`ConflictDetectionService` 为其门面）

###### 4.2.2.1 主流程 `detectAndPersistConflicts(String rir)`

| 项目 | 说明 |
|------|------|
| **功能** | 可选按 RIR 过滤，完成 **加载→过滤有效期→重置标记→双模块检测→去重→写 conflict_record→更新 rpki_cert 摘要→可选 Fabric 锚定**。 |
| **输入** | `rir == null` 表示全库。 |
| **输出** | `List<ConflictResult>`（去重后）。 |
| **核心逻辑** | 见下表逐步展开。 |
| **创新点** | **可配置** 父子判定、对等比较范围、AKI/SKI 约束；与链码 **谓词顺序对齐** 便于论文对比实验。 |

**步骤序列**：

1. `loadCerts(rir)`：`LambdaQueryWrapper` 按 RIR 或全表 `selectList`。  
2. `filterValidAt(all, now)`：UTC `now` 落在 `[notBefore, notAfter]`；缺省 null 边界时保守纳入。  
3. `resetConflictFlags(all)`：为本轮将涉及的汇总字段复位（具体实现见类内）。  
4. **`detectMultiHopChain`**：多跳撤销残留。  
5. **`detectPairwiseIndexed`**：IPv4 首字节桶 + `0.0.0.0/0` 全局桶，生成候选对，再 `detectIpPair`、`detectAsPair`、`detectIpAsBindingHeuristic`、`detectDirectRevokedPair`。  
6. `dedup`：去除重复 `ConflictResult`。  
7. `saveConflictRecords` / `updateCertConflictSummary`：持久化。  
8. **`fabricConflictAnchorService.anchorAfterDetectionIfEnabled(deduped, byId)`**：若 `fabric.anchor.auto-after-detect=true` 则锚定。

###### 4.2.2.2 关键私有方法

| 方法 | 功能 | 输入 | 输出 | 核心逻辑 | 作用 |
|------|------|------|------|----------|------|
| `walkAncestors` | 沿 `parent_cert_id` 上溯 | leaf、`byId` | 链列表 | 防环 `seen` | 多跳路径构造 |
| `detectMultiHopChain` | 祖先已撤后代仍有效 | valid、byId、out | void | 链上相邻对检查 `revoked` | **继承链一致性** |

| `hasValidityOverlap` | 两证有效期相交 | a、b | boolean | 区间不交判断 | 减少无效对 |

###### 4.2.3.3 `anchorAfterDetection`（链下检测→链上双批次）

| 项目 | 说明 |
|------|------|

| **创新点** | **双阶段上链**、**重试**、**应用侧审计与业务失败态分离**，直接支撑开题「一致性」「可审计」。 |

##### 4.4.4 形式化规则辅助方法（极度详细）

**（1）区间语义基础（内部类 `Prefix4` / `Prefix6`）**

- **`blockStartU32` / `blockEndU32`（IPv4）**：将 CIDR 映射为 **闭区间 \([S,E]\)** 上的 **无符号 32 位** 整数端点（`BigInteger` 防溢出）。
- **`cidrIntervalSubset(outer, inner)`**  
  - **形式化**：\(S_{outer} \le S_{inner}\) 且 \(E_{inner} \le E_{outer}\)，且 **`outer.plen ≤ inner.plen`**（外层前缀长度更短或相等才可能是超网）。  
  - **伪代码**：  
    ```
    if outer.plen > inner.plen: return false
    return S(inner) >= S(outer) && E(inner) <= E(outer)
    ```
  - **作用**：**包含关系**（父子资源单调性、包含型冲突）。

- **`cidrIntervalsIntersect(a, b)`**  
  - **形式化**：\(\max(S_a,S_b) \le \min(E_a,E_b)\)（闭区间相交非空）。  
  - **伪代码**：`return max(sa,sb) <= min(ea,eb)`。  
  - **作用**：**重叠判定** 的基础。

- **`rangesAdjacent(a, b)`**  
  - **形式化**：\(E_a+1 = S_b\) **或** \(E_b+1 = S_a\)（**无符号**意义下 +1）。  
  - **实现**：`ea.add(ONE).equals(sb) || eb.add(ONE).equals(sa)`。  
  - **作用**：**异长且不相交但边界紧贴** 的 **相邻歧义** 检测（开题「异长相邻」约束）。

**（2）`isPrefixContained`（IPv4/IPv6）**

- **逻辑**：`cidrIntervalSubset(a,b) || cidrIntervalSubset(b,a)`（**任一方覆盖另一方整块**）。  
- **在判定顺序中**：在 **`isPrefixOverlappingPartial`** 中 **先排除包含**，避免把包含误判为「部分重叠」。

**（3）`isPrefixOverlappingPartial`**

- **形式化**：相交 ∧ **非** `isPrefixContained`。  
- **逻辑**：若 `isPrefixContained` 则 false；否则 `cidrIntervalsIntersect`。

**（4）`isAdjacentDifferentLengthConflict`**

- **前置**：`plen(a) != plen(b)`；**非**包含且 **非**相交；最后 **`rangesAdjacent`**。  
- **意义**：捕获 **不同前缀长度** 下 **边界相邻** 的路由/授权歧义场景。

**（5）`hasSharedIpResource`**

- 双重循环：若存在 v4 或 v6 对 **`cidrIntervalsIntersect` 或 `isPrefixContained`**，返回 true。  
- **用途**：**AS 冲突前置**——仅在 IP 资源存在 **共享/重叠语义** 时才进入 AS 细判（与链下「共享 IP 前提」一致）。

**（6）`isASConflict`**

- **形式化**（注释）：设 \(S_a,S_b\) 非空，\(I = S_a \cap S_b\)。冲突当 **\(I=\emptyset\)** 或 **\(S_a \ne S_b\)**。  
- **实现**：`TreeSet` 交 + `emptyIntersection || !asA.equals(asB)`（在实现中与注释布尔式等价变换）。  
- **作用**：**同共享 IP 下的 AS 授权不一致**。

**（7）`collectInheritanceViolations`**

- **IPv4/IPv6**：子每个前缀须被父 **某一** 前缀 **`contains`（即 `cidrIntervalSubset`）**；若子有 IP 而父无对应族，记 **OUT_OF_SCOPE**。  
- **AS**：父子均非空 AS 时，子每个 ASN 必须 **∈ 父集合**。  
- **返回**：固定顺序的字符串行，便于 **`appendInheritanceFindings`** 写 raw+log。

**（8）`runPairDetection` 内执行顺序（代码即法律）**

1. **`runInheritancePhase`**（继承）  
2. **`checkASConflictOnSharedResource`**（共享 IP 前提下的 AS）  
3. **`checkV4PrefixRules`**：**对每个 (pa,pb)** 顺序：**包含 → 部分重叠 → 异长相邻**  
4. **`checkV6PrefixRules`**：同序  
5. 构造 **`FindingEntry`**（**`rulePhase`**：`inferRulePhase`）  
6. **`worstConflictType`** 取枚举 **ordinal** 最大者  
7. 若非 `NONE`，**`buildEvidenceAsset`** + `writeConflictEvidenceBatch`（在 detect 路径由调用方传入 batch）

##### 4.4.5 可审计性

- **`LOG.info("[AUDIT][tx=...][phase=...][pair=...][rule=...] ...")`**：细粒度规则命中。  
- **`FindingEntry`**：`type`、`detail`、`certHashA/B`、`conflictScope`、`involvedCertHashChild`、**`rulePhase`**。  
- **应用侧**：**`correlation_id` + `pipeline_step`** 与 **`[AUDIT]`** 互补——前者在 **业务库** 串联轮次，后者在 **链码日志** 串联模拟步骤。

#### 常见问题排查

| 现象 | 处理 |
|------|------|
| 私钥文件找不到 | 更新 **`key-path`** 为实际 `*_sk`。 |
| `fabric_send_failed=true` | 查 **`fabric_last_error`**、`fabric_tx_record`；观察是否 **`retry-max`** 次后仍失败。 |
| 检测后无链上交易 | **无冲突** 时设计为 **不提交**；或 **`auto-after-detect=false`**。 |
| 背书不一致 | 使用 **双组织 connection profile**；链码需 **`getTxTimestamp`** 版本。 |
| 误扫项目内 `.cer` | 确认 **`rpki.bulk-import.enabled=false`**。 |
| Wallet 身份错误 | 清理 wallet 按文档重新导入 Admin。 |

### 7. 系统验证与使用示例

#### 7.1 端到端流程（链下检测 → 自动存证 → 查询）

1. **导入**：`POST /api/cert/bulk-import-directory` 或 `POST /api/cert/download-and-parse`。  
2. **检测**：`POST /api/cert/detect-conflicts`（若 `fabric.anchor.auto-after-detect=true` 则自动锚定）。  
3. **查库**：`SELECT has_conflict,fabric_tx_id,is_sent_to_fabric,fabric_send_failed FROM rpki_cert WHERE has_conflict=1 LIMIT 10;`  
4. **查链**：`GET /api/fabric/chain/conflicts`；`GET /api/fabric/chain/audit/{certHash}`。

#### 7.2 典型 curl

```bash
curl -s http://localhost:8082/api/
curl -s http://localhost:8082/api/fabric/chain/conflicts
curl -s "http://localhost:8082/api/fabric/chain/audit/<certHash>"
curl -s -X POST http://localhost:8082/api/cert/detect-conflicts -H "Content-Type: application/json" -d '{}'
curl -s -X POST http://localhost:8082/api/fabric/chain/detect-on-chain/anchor-db \
  -H "Content-Type: application/json" -d @detect-result.json
```

#### 7.3 审计记录查看

- **应用侧**：`SELECT * FROM fabric_tx_record WHERE correlation_id='<uuid>' ORDER BY id;`  
- **账本侧**：`GET /api/fabric/chain/audit/{certHash}`；peer/链码日志中检索 **`[AUDIT]`**。
