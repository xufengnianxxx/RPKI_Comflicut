# 测试数据编造清单（`rpki_cert` 字段改写说明）

## 1. 条数说明：为什么是 50 条而不是 59 条？

本仓库 **`test-conflict-construction/01_pick_50_and_backup.sql`** 与 **`02_apply_mutations.sql`** 的设计是：

- 从数据库中**随机抽取 50 张**证书，写入辅助表 **`rpki_cert_conflict_pick`**，槽位 **`slot = 1 … 50`**；
- 再对这 **50 行** `rpki_cert` 按槽位**改写资源字段、父子关系、撤销标记**等。

因此，在**标准脚本**下，被「编造/改写」的是 **50 条** `rpki_cert` 记录，**不是 59 条**。

若您记忆中是「59 条」，可能来自：

- 其它环境**手工多改过**若干行；
- 把 **`conflict_record` 里生成的冲突条数**（例如某次检测跑出 50+ 条）与「改库条数」混在一起；
- 或把 **`backup_before_test.sql` 里 `conflict_record` 的插入条数**（该文件里为 **52 条**冲突记录）与其它数字相加。

**结论**：本文以 **`02_apply_mutations.sql` 为准**，共 **50 个槽位**；**具体主键 `id` 不固定**，需用下文 SQL 在您当前库中查询。

## 2. 如何查询「当前」每个槽位对应的 `id` 与 `cert_hash`

在已执行 `01` + `02` 之后运行：

```sql
USE rpki_db;
SELECT slot, id, LEFT(cert_hash, 16) AS hash16, parent_cert_id,
       LENGTH(ipv4_prefixes) AS len_v4, LENGTH(ipv6_prefixes) AS len_v6, LENGTH(as_numbers) AS len_as, revoked
FROM rpki_cert_conflict_pick p
JOIN rpki_cert c ON c.id = p.id
ORDER BY slot;
```

若只关心 id 列表：

```sql
SELECT slot, id FROM rpki_cert_conflict_pick ORDER BY slot;
```

- **`cert_hash` / `raw_cert_data` / `issuer` / `subject` 等**：脚本**未**为测试而改写；若检测或上链依赖哈希一致性，请注意「只改 JSON 资源字段」与「原始 DER 指纹」是否仍一致——**链下读库**与**链上读摘要**需分别理解。

## 4. 与 `qidongjiaoben/backup_before_test.sql` 的关系

该文件是某次环境的 **MySQL dump**，其中包含 **`conflict_record` 52 条**插入语句（检测**输出**），**不是**「50 槽位改库」的 id 清单。

若需从该备份中查看当时涉及的证书 id，可检索 `cert_id_a`、`cert_id_b` 列（例如 id **1250**、**1252** 等父证书占位），与本文 **slot 机制**无直接一一对应关系。

---

## 5. 若必须坚持「59 条」——建议核对步骤

1. `SELECT COUNT(*) FROM rpki_cert_conflict_pick;` —— 应为 **50**。  
2. `SELECT COUNT(*) FROM rpki_cert WHERE updated_at > '<您执行02的时间>';` —— 自行判断是否多出 9 条。  
3. 将多出的 id 导出后，可追加到本清单第二节表格下方作为「补充手工改写」。
