## 2. 性能测试方法

### 2.1 工具与方法

由于本机当前缺少可直接执行的 JMeter/Caliper 运行条件（见第 4 节“环境限制”），本次采用可复现实测方案：

1. 使用 Python 并发请求进行轻量压力采样；
2. 每个接口发送固定请求数，统计 `RPS / 平均时延 / P95 / P99 / 最大时延 / 成功率`；
3. 作为 Chapter 4 的“基线性能结果”。

### 2.2 测试接口与负载

| 接口 | 方法 | 并发 | 请求数 | 说明 |
|---|---|---:|---:|---|
| `/api/` | GET | 1 | 50 | 健康检查基线 |
| `/api/cert/certs?page=1&size=15` | GET | 10 | 100 | 证书分页查询 |
| `/api/cert/pair-detection-records?page=1&size=15` | GET | 10 | 100 | 检测记录分页查询 |

## 4. 环境限制与可复现说明

### 4.1 当前限制

1. `jmeter` 命令未安装（系统提示需 `apt install jmeter`）。
2. Caliper 依赖未安装完成，执行 `npm --prefix caliper run test` 报错 `could not determine executable to run`。
3. 当前在线服务未启用 `POST /api/test/functional`，因此性能压测未覆盖该接口。

### 4.2 完整压测复现（环境补齐后）

```bash
# 1) 安装 caliper 依赖
cd caliper && npm install && cd ..

# 2) 运行 Caliper
npm run caliper:test

# 3) 运行 JMeter
jmeter -n -t qidongjiaoben/jmeter/RPKI-Conflict-Detection-Test-Plan.jmx \
  -l qidongjiaoben/jmeter/result.jtl \
  -e -o qidongjiaoben/jmeter/report

# 4) 启动系统资源监控
./qidongjiaoben/monitor-performance.sh qidongjiaoben/perf-data 2 60
```

## 5. 结果分析

1. 查询类接口在并发 10 下表现稳定，成功率 100%，无 5xx 或超时。  
2. `list_certs` 的时延显著高于 `pair_records`，说明证书分页查询的数据库扫描/序列化成本更高。  
3. 在当前轻量压力下，系统吞吐位于 140~257 req/s 区间，具备演示级与中小规模实验可用性。  
4. 后续优化建议：
   - 为高频列表接口增加索引与投影字段；
   - 对热点分页加入缓存；
   - 对写路径（`detect-pair-persist`）考虑异步化和批量写。
