# 论文第4章草稿：系统测试与性能评估

## 4.1 测试目标

本章目标是验证系统在“功能正确性 + 性能可用性”两个维度均满足毕业设计要求：

1. 功能层面：12 类冲突场景可稳定复现，检测结果与预期一致。
2. 性能层面：在中等并发下 REST 服务吞吐与时延可接受，且可复现实验。

## 4.2 功能测试设计

- 测试数据：`TestDataGenerator` 自动生成 `qidongjiaoben/test-data/`
- 测试入口：
  - REST: `POST /api/test/functional`，`GET /api/test/functional/report`
  - CLI: `--test-mode=functional`
- 输出文件：`qidongjiaoben/测试结果-功能测试.md`

### 4.2.1 用例覆盖

覆盖 D1-D12（IPv4/IPv6 包含、重叠、邻接、父子越权、撤销链、AS 冲突），共 12 条。

### 4.2.2 功能测试结果

根据当前执行结果，12/12 用例通过，关键修复点包括：

1. AS 扩展解析：修正 RFC 3779 `asIdentifiers` 解析逻辑；
2. TC07/TC08 桶与越权边界修正；
3. TC09 证书 AS 字段由空恢复为 `[65000]/[65001]`。

## 4.3 性能测试设计

### 4.3.1 Caliper

构建 `caliper/` 目录，定义 4 组负载：

1. `GET /api/test/functional/report`
2. `POST /api/test/functional`
3. `GET /api/cert/certs`
4. `POST /api/cert/detect-pair-persist`

通过 `npm run caliper:test` 一键执行。

### 4.3.2 JMeter

构建 `qidongjiaoben/jmeter/RPKI-Conflict-Detection-Test-Plan.jmx`，包含 2 个线程组：

1. 功能接口压力（10 用户）
2. 双证检测压力（10/50/100 用户）

### 4.3.3 资源监控

使用 `qidongjiaoben/monitor-performance.sh` 按固定间隔采样 CPU、内存、网络、Docker、MySQL，输出 CSV 作为实验原始证据。
